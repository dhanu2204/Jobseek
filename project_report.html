JobSeek: Career Assistant Platform

Complete Technical Project Architecture & Functionality Documentation

1. Executive Summary

JobSeek is a comprehensive AI-powered career enhancement system designed to bridge the gap between job candidates and industry recruiters. The platform comprises a React frontend built with Vite and a Spring Boot (Java) backend connected to a MySQL relational database. The core feature set includes a dynamic Resume Builder, an Automated Tracking System (ATS) Resume Matcher, and an AI-driven Mock Interviewer powered by the Google Gemini model. Additionally, the platform integrates a premium tier restriction layer backed by the Razorpay payment gateway to monetize resources.

2. Architecture & Tech Stack

The project leverages a modern, decoupled client-server architecture:

  • Frontend: Built with React (Vite). Uses standard CSS for styling and custom modules for features like markdown parsing, Speech Recognition (Web Speech API), and text-to-speech synthesis.
  • Backend: Developed in Spring Boot (v4.1.0), using Spring MVC for REST APIs, Spring Data JPA for object-relational mapping, and Lombok for boilerplate reduction.
  • Database: MySQL (v8.0), maintaining schemas for users, uploaded resumes, applied jobs, completed challenges, and interview scorecards.
  • AI Services: Powered by Google Gemini 2.5 Flash via HTTP REST interfaces to generate scores, critiques, and questions.
  • Payments: Razorpay Java SDK and Frontend SDK for transaction processing.

3. Database Models & Schema

The database consists of five main relational tables mapping the application entities:

A. Users Entity (users table)

Maintains account credentials, profiles, premium status, and scan usage counts.

B. Resume Entity (resume table)

Stores details for candidate resumes, structured in JSON arrays for skills, experience, projects, and education.

C. MockInterviewReport Entity (mock_interview_report table)

Stores detailed performance scorecard evaluations returned by the AI interviewer, keeping history for candidates.

4. Core Platform Functionalities

A. Resume Builder

The Resume Builder gathers personal information, headlines, detailed experiences, and skills using a multi-step form. Once submitted, the data is packaged into a JSON payload and sent to the backend, enabling candidates to format and render standardized professional resumes.

B. ATS Resume Matcher

The ATS Matcher compares a candidate's resume (PDF or plain text) against a target job description:

  1. Extracts text from the job description and candidate resume.
  2. Constructs an advanced analysis prompt and queries the Gemini 2.5 Flash model.
  3. Returns a clean JSON response containing a match score, a list of missing keywords, and detailed bullet-point optimization suggestions.

C. Zoro Mock Interviewer (Speech & AI System)

The mock interviewer behaves like a live, professional technical recruiter named Zoro:

Key Features:
  • First Question Greeting: Bypasses API loading delay. Starts immediately with a voice greeting: "Hello! I am Zoro... Can you please introduce yourself?"
  • Persistent Speech Recording: Uses the browser's Web Speech API. If a candidate pauses to think, the system handles the no-speech error code and restarts recording after a 800ms delay. It appends new sentences cleanly to previous sentences rather than overwriting them.
  • Male Voice Utterance: Plays Zoro's questions using a male text-to-speech engine filter (e.g. Microsoft David).
  • Difficulty Scenarios:
    • Beginner: Evaluates only skills, asking core programming questions.
    • Intermediate: Mixes core concepts, framework internals, and basic project details.
    • Hard: Deep-dives into concurrent operations, system design, and the candidate's exact project tech stacks.

5. Premium Monetization & Payment Flow

The system limits free users to 100 Resumes, 100 ATS Scans, and 100 Mock Interviews. Reaching any limit blocks access and displays a premium pop-up modal redirecting to the payments page.

[Candidate] ---> Click "Pay Premium" ---> [Spring Boot: create-order] | Calls Razorpay API | [Razorpay Popup] <--- returns Order ID <-----------+ | Candidate completes UPI/Card Payment | [Spring Boot: verify-payment] ---> HMAC-SHA256 Check Signature | [DB: users table] <--- Upgrade Premium <-+

A. Backend Integration

The backend verifies Razorpay's SHA-256 webhook/checkout signatures to guarantee that payment state upgrades cannot be forged: 

Mac sha256 = Mac.getInstance("HmacSHA256");
SecretKeySpec secretKey = new SecretKeySpec(keySecret.getBytes(), "HmacSHA256");
sha256.init(secretKey);
byte[] hash = sha256.doFinal(data.getBytes());
String generatedSignature = Hex.encodeHexString(hash);

6. Code Flow & Flaw Corrections

During construction, several critical technical issues were resolved:

  • React ReferenceError (Home.jsx): The page initially crashed after login because `useEffect` was not imported. This was resolved by importing `useEffect` from React.
  • Jackson Deserialization Primitive Mismatch: The backend crashed with `400 Bad Request` during login and registration when mapping null values to primitive Java types (`boolean` and `int`). Mappings were changed to wrapper objects (`Boolean` and `Integer`) to allow null fields cleanly.
  • SQL Constraint Violation: The `users` table columns had a `NOT NULL` restriction. This caused registration to fail since the signup form did not supply counts. Safe default values (`false` and `0`) were explicitly assigned in the `registerUser` service method to solve this.
  • Microphone Stale Closure: Web Speech API event listeners locked themselves to initial empty strings due to stale closures. Resolving this involved declaring a mutable tracking ref (`voiceTranscriptRef`) to keep text synced and appending sentences across sessions.
